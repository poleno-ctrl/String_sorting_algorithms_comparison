## Теоретическая сложность и эмпирические результаты

### ID посылок:

**A1m - Реализация STRING MERGE SORT**     - 375756101

**A1q - Реализация STRING QUICK SORT**     - 375758602

**A1r - Реализация MSB RADIX SORT**        - 375760027

**A1rq - Реализация MSB RADIX+QUICK SORT** - 375760102

### Обозначения

- $N$ — количество сортируемых строк.
    
- $L$ — средняя длина строки.
    
- $D$ — суммарная длина различающих префиксов (в худшем случае $D \approx N \cdot L$).
    
- $\Sigma$ — мощность алфавита.
    

### Теоретическая асимптотика

| **Алгоритм**                | **Временная сложность (в среднем)** |
| --------------------------- | ----------------------------------- |
| **Std QuickSort**           | $\mathcal{O}(L \cdot N \log N)$     |
| **Std MergeSort**           | $\mathcal{O}(L \cdot N \log N)$     |
| **String QuickSort**        | $\mathcal{O}(N \log N + D)$         |
| **String MergeSort**        | $\mathcal{O}(N \log N + D)$         |
| **MSD Radix (No Fallback)** | $\mathcal{O}(N \cdot \Sigma + D)$   |
| **MSD Radix (Fallback)**    | $\mathcal{O}(N \log N + D)$         |

### **Эмпирический анализ**

**String QuickSort (3-way)**

Алгоритм показал наилучшее время. Теоретически оптимален: выполняет минимум необходимых посимвольных сравнений (`O(D)`), избегая повторных проверок префиксов.

**MSD Radix Sort (Fallback)**

Алгоритм показал второе время. Выполняет крайне малое число посимвольных сравнений (они происходят только при переключении на QuickSort для коротких подмассивов). Гибридизация устраняет накладные расходы памяти.

**Std QuickSort**

Алгоритм показал среднее время. Теоретически неоптимален: совершает избыточное число посимвольных сравнений (`O(L * N log N)`), впустую сравнивая уже проверенные совпадающие префиксы.

**MSD Radix Sort (No Fallback)**

Алгоритм показал время ниже среднего. **Количество посимвольных сравнений строго равно 0**. Алгоритм использует значение символа напрямую как индекс массива (корзины). Сильно проигрывает по времени из-за выделения памяти под пустые корзины на малых подмассивах.

**String MergeSort (LCP)**

Алгоритм показал худшее время. Теоретически оптимален по числу посимвольных сравнений (сравнивает только `O(D)` символов благодаря массиву LCP), но проигрывает из-за постоянной аллокации памяти в куче при слиянии.

**Std MergeSort**

Алгоритм показал худшее время. Теоретически неоптимален: совершает максимум избыточных посимвольных сравнений (`O(L * N log N)`) при слиянии и дополнительно страдает от высоких затрат на аллокацию памяти.
